9 Storm detection and tracking algorithm
10 =========
11 This library provides methods for searching latitude-longitude data sets
12 in [NetCDF](http://www.unidata.ucar.edu/software/netcdf/) format for various kinds of storms which may be defined by the user. @n
13 Storm identification criteria may correspond to any meteorological phenomena of interest that can be described quantitatively
14 in terms of available data.
15 
16 The algorithm reads the input data file and at each time step completes a spatial search, outputing locations that meet
17 or exceed the spatial identification criteria.@n
18 Following the spatial search, detected storms are built into tracks using a temporal correlation program, which also applies
19 temporal identification criteria.
20 
21 Spatial search
22 ---------------
23 The user must specify a sector radius (in km) to define the search sectors. @n
24 Only one storm may be detected per radius so this input is a tunable parameter of the algorithm. @n
25 Depending on the application, larger or smaller radii may be desired. @n
26 Any 2 storms that are separated by a great-circle distance less than the sector radius will be considered duplicates and one of them will be removed.
27 
28 The spatial search algorithm:
29 * Covers the requested search area (specified by a maximum and minimum latitude) with overlapping circular search sectors,
30 defined by the user-specified great-circle distance in km.
31 * Reads the NetCDF input file for relevant storm data (i.e., windspeed, sea level pressure, vorticity) in each storm sector.
32 * Stores information about any sector (and its relevant data) in a linked-list data structure if that sector's data
33 exceeds some user-defined storm threshold.
34 * After all sectors have been searched, it compares the found storms from the linked-list to remove duplicates.
35 * Outputs the linked-list to various formats.
36 
37 Temporal correlation
38 -----------------
39 The temporal correlation program takes the output of all per-timestep Spatial Searches and examines the data for correlations
40 across adjacent time steps to build storm tracks. @n
41 Users must define a maximum storm speed (m/s). @n
42 This is used to determine the maximum distance a storm can travel per timestep.
43 
44 * Beginning at the first timestep containing detected storms in the output of the spatial search procedure,
45 the tracking program starts with the first detected storm and searches subsequent timesteps for nearest neighbors.
46 * Successive locations are stored in a possible track.
47 * Temporal identification criteria (e.g., minimum duration, distance traveled over time, etc.) are applied to each possible track.
48 * If the temporal criteria are met or exceeded, the possible track is saved to output.
49 * The search begins again with the next storm detection from the spatial search output, and continues until all data
50 has been searched.

52 Software requirements
53 ===============
54 The software requires [NetCDF](http://www.unidata.ucar.edu/software/netcdf/) and [NetCDF Fortran](http://www.unidata.ucar.edu/software/netcdf/)
55 libraries and a modern Fortran compiler. @n
56 An MPI distribution is not required. @n
57 
58 NetCDF file access utilities provided by the __Geophysical Fluid Dynamics Laboratory's (GFDL)__
59 [TSTORMS](http://www.gfdl.noaa.gov/tstorms) code for tropical cyclone detection are incorporated into this software.

98 Software / algorithm design
99 ===============
100 The fundamental data structure in the StrideSearch software is a linked-list e.g. @ref StormListNode, @ref TrackListNode. @n
101 We do not know in advance how many storms will be detected per timestep, or how many storm tracks may be built from one dataset. @n
102 Linked-lists handled these unknowns using dynamic memory allocation; each time a new storm or new track is found, the software
103 allocates memory for that storm and saves its information to that newly allocated memory. @n
104 The basic structure of the linked-lists used by StrideSearch is
105 
106  type ListNode
107  real :: aStormDatum
108  integer :: anotherStormDatum
109 
110  type(ListNode), pointer :: nextNode
111  end type
112 
113 Users may alter the data items of a list node (StrideSearch has an example of this: compare @ref StormListNode to @ref TropicalStormListNode). @n
114 The pointer `nextNode` is the means of constructing and traversing a list. @n
115 It points to the location in memory of the next node or, if it is the end of the list, it is a null pointer. @n
116 Both the spatial search algorithm (@ref StrideSearch ) and the temporal correlation algorithm (@ref TrackListNode) use this basic data structure.
117 
118 Lists are traversed with other pointers. @n
119 A pointer, say `current`, may point to the current node and be used to access its data.@n
120 Then the entire list may be traversed with code similar to the following
121 
122  type(ListNode), pointer :: current
123 
124  current => listRoot
125  do while ( associated(current) )
126  ! < do something with the data at current node >
127  current => current%nextNode
128  enddo
129 
130 This do loop will automatically terminate when `current` reaches the null pointer `current%%nextNode` at the end of the list.
131 
132 This basic data structure is applied to each application. Each application has three files (and three data types) associated with it.
133 1. A data file. StrideSearchData provides the basic data container for netcdf data to be searched. This data type may
134 be extended to facilitate other applications. Examples are provided by TropicalData and PolarData.
135 2. A storm list node. The linked list of storms per-timestep that will be output by the search algorithm. The basic type is
136 given by StormListNode, and example extensions may be found in TropicalStormListNode and PolarLowListNode.
137 3. A driver program. Examples are given by PolarSearch.f90 and TropicalSearchDriver.f90.
138 
139 Use
140 ===============
141 Users may extend the basic @ref StormListNode data type following the example of @ref TropicalStormListNode to an application of their choosing. @n
142 Each linked list relies on the `initialize` and `Copy` subroutines to construct linked lists, and must also implement
143 `removeNodeFromList` and recursive `deleteList` procedures. @n
144 The @ref StrideSearchData class and subroutines should also be extended or altered to read the data corresponding to the new application. @n
145 Finally, the @ref stridesearchmodule::dostridesearch subroutine should be modified to include the storm identification criteria suitable for the new application. @n
146 An example of each of these extensions of the basic Stride Search data types and methods are provided by @ref TropicalData and @ref TropicalStrideSearch.
